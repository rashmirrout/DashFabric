# CB (ControllerBridge) Implementation Plan

> **Owner:** CB Team (integrators)
> **Scope:** Phase 1 (proto delivery, config pipeline, event streaming, FM integration)
> **Timeline:** 16 weeks (4 sprints × 4 weeks, parallel with FM)
> **Status:** Ready to start
> **Dependency:** Coordinates with FM; not blocking (parallel development)

---

## Executive Summary

CB Phase 1 implementation focuses on config pipeline: (1) proto schemas → (2) data ingestion → (3) event streaming (etcd/Kafka) → (4) FM integration → (5) end-to-end validation.

**Total effort:** ~300 engineer-days (3 FTE × 16 weeks)

**Critical path:** Proto schemas (week 1) → data ingestion (weeks 2–5) → event streaming (weeks 6–10) → FM integration (weeks 11–14) → E2E testing (weeks 15–16).

### Sprint 3: Event Streaming & Pub/Sub (Weeks 9–12)

**Goal:** Implement event streaming to FM; support both etcd and Kafka backends.

#### Tasks

| Task | Description | Duration | Acceptance Criteria |
|------|-------------|----------|---|
| **3.1** | Design pub/sub abstraction (backend-agnostic) | 3 days | Interface: Publish(), Subscribe(), Unsubscribe() |
| **3.2** | Implement etcd backend (watch-based streaming) | 5 days | Streams config changes via etcd watch; handles reconnection |
| **3.3** | Implement Kafka backend (topic-based streaming) | 5 days | Produces messages to Kafka topics; supports compacted topics |
| **3.4** | Implement topic naming schema | 2 days | `/dashfabric/v1/config/vnets/<vnet_id>`, `/config/nics/<eni_id>`, etc. |
| **3.5** | Implement message encoding (protobuf + JSON fallback) | 3 days | Messages encoded as protobuf; fallback to JSON if needed |
| **3.6** | Implement replay logic (FM catches up on startup) | 4 days | FM can replay all events from compacted topic on startup |
| **3.7** | Implement dead-letter queue (failed messages) | 2 days | Messages that FM rejects go to DLQ; operator can inspect |
| **3.8** | Implement heartbeat + keepalive (connection health) | 3 days | CB sends periodic heartbeat; FM expects one every 30s |
| **3.9** | Write integration tests (etcd + Kafka) | 6 days | >95% coverage; tests: pub/sub, replay, failure recovery |
| **3.10** | Performance test: 10k events/sec streaming | 3 days | Stream 10k config updates/sec without loss |

**Sprint 3 Output:**
- ✅ Event streaming to FM (etcd + Kafka support)
- ✅ Pub/sub abstraction (pluggable backends)
- ✅ Message replay on FM startup
- ✅ Dead-letter queue for failed messages
- ✅ 40+ integration tests
- ✅ Performance test passing (10k events/sec)
- **Artifact:** `pkg/streaming/`, `pkg/pubsub/`

### Sprint 4: FM Integration & E2E Testing (Weeks 13–16)

**Goal:** Full CB ↔ FM integration; E2E validation of 6 routing constructs.

#### Tasks

| Task | Description | Duration | Acceptance Criteria |
|------|-------------|----------|---|
| **4.1** | Define CB ↔ FM contract (error codes, retry logic) | 2 days | Document error handling; CB knows what FM rejects |
| **4.2** | Implement CB error handling (FM sends error responses) | 3 days | CB listens for FM errors; logs + alerts |
| **4.3** | Implement retry logic (exponential backoff) | 2 days | CB retries failed config sends; max 5 retries; 1s–32s backoff |
| **4.4** | Implement config versioning + ordering guarantees | 3 days | CB tracks config version; ensures FM sees updates in order |
| **4.5** | Write E2E test: peering (vnet-acme ↔ vnet-shared) | 3 days | CP → CB → FM; ENI in both VNETs can forward to each other |
| **4.6** | Write E2E test: VIP (backend + NAT) | 3 days | CP → CB → FM; VIP backend add triggers ENI re-hydration |
| **4.7** | Write E2E test: meter (policy + traffic class) | 3 days | CP → CB → FM; metered traffic respects CIR/PIR |
| **4.8** | Write E2E test: Private Link (PE mapping) | 3 days | CP → CB → FM; PE endpoint discovered via routing_action_hint |
| **4.9** | Write E2E test: direction (inbound + outbound) | 3 days | CP → CB → FM; inbound/outbound routes validated separately |
| **4.10** | Write E2E test: Service Tunnel (tunnel + SNAT) | 3 days | CP → CB → FM; service tunnel routes with per-route SNAT |
| **4.11** | Write E2E failure recovery (config rollback, retry) | 4 days | CP sends invalid config; FM rejects; CB retries; eventually succeeds |
| **4.12** | Write E2E load test (1k concurrent ENI provisions) | 4 days | CB streams 1k ENI configs; FM hydrates all in <30s |
| **4.13** | Write documentation (integration guide, operator manual) | 4 days | README, config examples, troubleshooting guide |
| **4.14** | Security review (authentication, encryption, validation) | 3 days | CB validates all inputs; encrypts in-flight data if needed |

**Sprint 4 Output:**
- ✅ Full CB ↔ FM integration working
- ✅ Error handling + retry logic
- ✅ 6× E2E tests (one per routing construct) passing
- ✅ E2E failure recovery tests passing
- ✅ Load test (1k concurrent provisions < 30s)
- ✅ Complete documentation + operator manual
- **Artifact:** `tests/e2e/`, `docs/integration/`

## Critical Path & Dependencies

```
Sprint 1: Proto finalization
    ↓
Sprint 2: Data ingestion (depends on protos from sprint 1)
    ↓
Sprint 3: Event streaming (parallel with sprint 2)
    ↓
Sprint 4: FM integration (depends on FM being ready; parallel stream)
          E2E tests (depends on both CB sprint 3 + FM all sprints)
```

**Critical path:** Protos → ingestion → streaming → FM integration → E2E tests.

**Parallel tracks:** CB sprints 2–4 can proceed in parallel with FM sprints 1–4 (minimal blocking).

## Risks & Contingency

| Risk | Probability | Impact | Mitigation |
|------|-------------|--------|---|
| Proto schema changes mid-sprint | Medium | Medium | Lock protos by end of sprint 1; document change process |
| Etcd watch semantics misunderstood | Low | High | Spike on etcd watch behavior early (week 2) |
| Event ordering issues (updates out of order) | Medium | High | Implement version tracking + ordering checks in CB |
| Performance target (10k events/sec) missed | Medium | Medium | Profile streaming + optimize serialization |
| FM not ready by sprint 4 | Low | High | Use mock FM (test harness) for E2E testing |
